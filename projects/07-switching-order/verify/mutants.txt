# One mutant per real behaviour the code claims to have. Every one MUST make the suite go red. A
# survivor is a finding and it is one of three things: dead code, a coverage hole, or an EQUIVALENT
# mutant that no input can distinguish (pattern 14). Prove equivalence before claiming it.

# ⚠ RUNNER CONSTRAINT: mutants.txt is parsed with `IFS='|' read`, so A SED EXPRESSION MAY NOT CONTAIN

# score a kill it never earned. Express "any one of these suffices" by dropping each term in turn.

# 07'S MUTANTS COME IN FOUR FLAVOURS.
#
#   1. BREAK THE TOPOLOGY. The model is traced from two figures and the whole project rests on it
#      being right. These move a blade or delete an edge, and BPA's published answers stop
#      reproducing. Flavour 1 is the answer to "how do you know you read the drawing correctly" --
#      not fully, but a wrong blade cannot satisfy two examples that share one figure.
#   2. BREAK THE ALGORITHM. Zone growth, boundary detection, source reachability, the verbs.
#   3. BREAK THE RULE CHECKER. IV.4.C is the gate; a gate that cannot fail is decoration.
#   4. 🔴 BREAK NOTHING, AND ONLY MAKE THE TOOL CLAIM MORE THAN ITS SOURCE. Section F's mutants
#      compute nothing wrong. They let the tool call a generated sequence BPA's, or assume a breaker
#      can interrupt what flows through it. This repo's failure mode is not a broken feature, it is
#      an honest overclaim, and flavour 4 is the only one that looks for it.

# 1. THE TOPOLOGY. Traced from the figures at 14x. If these do not go red, the tracing is untested.

# 🔴 THE ONE THAT MATTERS MOST. The repo's own SOURCES.md described the Example 1 figure as "PCB A-10

build/topology.js | /id: 'a10_line_side_disc'/,/},/ s/state: 'closed'/state: 'open'/ | A-10 Line Side Disconnect starts open (BPA says "open and tag" it, so it was closed)

# ---------------------------------------------------------------------------------------------
# 2. THE ALGORITHM.
# ---------------------------------------------------------------------------------------------

# A BREAKER IS NOT AN ISOLATING DEVICE. That single line is most of the domain: BPA opens PCBs to
# de-energize and STILL requires disconnects opened and tagged as the limits. Stop the zone crossing
# the breaker and Example 2's answer (which spans it) falls apart.
build/engine.js | s/const ZONE_TRAVERSABLE = \['pcb', 'fixed'\];/const ZONE_TRAVERSABLE = ['fixed'];/ | zone no longer grows across a breaker

# The inverse: let the zone grow through disconnects too and every boundary dissolves.
build/engine.js | s/const ZONE_TRAVERSABLE = \['pcb', 'fixed'\];/const ZONE_TRAVERSABLE = ['pcb', 'fixed', 'isolating'];/ | zone grows through isolating devices (there are no limits left)

# 🔴 LIMITS ARE TOPOLOGICAL, NOT STATEFUL, AND BPA PROVES IT TWICE (Example 2's already-open Line
# Side Disconnect; Example 5's "open low voltage isolating device ACB-1"). Make the source search
# respect the live state and Example 2's answer collapses to a single device.
build/engine.js | s/if (d.kind !== 'isolating') continue;/if (d.kind !== 'isolating') continue; if (d.state === 'open') continue;/ | limits made state dependent (an already-open device stops counting as a limit)

# Ground is not a source, so a ground switch is never a limit. Let the search walk through one and
# the ground switches start showing up as clearance limits, which BPA never does.
build/engine.js | /function reachesSourceAvoidingZone/,/^}/ s/if (d.kind === 'ground') continue;/if (d.kind === 'nonexistent') continue;/ | ground switches traversed as if ground were a source

# The boundary test. Both endpoints in, or both out, means not a boundary.
build/engine.js | s/if (aIn === bIn) continue;/if (aIn \&\& bIn) continue;/ | boundary test accepts devices wholly outside the zone

# Re-entering the zone is not an energization path TO the zone.
build/engine.js | s/zone\.has(far) .. seen\.has(far)/seen.has(far)/ | source search allowed to path back through the zone

# The verbs. BPA says "check open" for an open device and "open" for a closed one.
build/engine.js | s/return device.state === 'open' ? 'check open and tag' : 'open and tag';/return device.state === 'open' ? 'open and tag' : 'check open and tag';/ | tag verbs inverted

# energized() is the stateful half and is what shows the feeder live through the bypass.
build/engine.js | s/if (d.state !== 'closed') continue;   \/\/ an open device conducts nothing/if (d.state === 'nonexistent') continue;/ | open devices conduct (nothing is ever dead)

# ---------------------------------------------------------------------------------------------
# 3. THE RULE. BPA S-6 IV.4.C: "PCB(s) shall be checked open before operating isolating devices."
# ---------------------------------------------------------------------------------------------

# The gate itself. If reversing the sequence does not fail, the gate is decoration.
build/engine.js | s/return { ok: violations.length === 0, violations, rule: RULE_IV_4_C };/return { ok: true, violations: [], rule: RULE_IV_4_C };/ | IV.4.C checker always passes

# A disconnect that is only CHECKED open is not OPERATED, so it does not engage the rule. Treat it as
# operated and the rule fires on sequences BPA's own example would fail.
build/engine.js | s/const firstIsolatingOp = steps.findIndex((s) => s.kind === 'isolating' \&\& s.op !== 'check open and tag');/const firstIsolatingOp = steps.findIndex((s) => s.kind === 'isolating');/ | checking a disconnect open counted as operating it

# The breakers must be emitted before the disconnects, which is the rule expressed as an order.
build/engine.js | s/for (const pcbId of pcbsToCheckOpen(topo, workItem)) {/for (const pcbId of []) {/ | no breaker steps emitted at all

# A disconnect already open is checked open, not operated, so it does not drag a breaker in with it.
build/engine.js | s/if (limit.state === 'open') continue;   \/\/ checked open, not operated: the rule is not engaged/if (false) continue;/ | already-open limits treated as operations (pulls in breakers BPA does not open)
